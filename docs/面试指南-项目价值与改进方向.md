# 面试指南：AI 推理网关

## 一、项目一句话描述

> 用 C++ 从零搭建的高并发 AI 推理服务网关——把单实例模型封装成支持动态批处理的 HTTP 后端，实测吞吐提升 8 倍。

---

## 二、项目的核心价值（技术面叙事）

### 2.1 解决了什么问题

单实例 AI 模型一次只能推理一个输入。如果直接给客户端调用，并发 10 个请求就是 10 次独立推理 = 10 倍延迟。**动态批处理**把分散的请求攒成一批，一次推理完成全部——延迟不变，吞吐翻倍。

### 2.2 技术亮点（面试展开用）

| 技术点 | 为什么有价值 | 你怎么做的 |
|---|---|---|
| **动态批处理调度** | 核心创新，区别于静态 batch | 三级优先级队列 + 积累窗口 + 时间窗口攒 batch |
| **线程间同步** | 展示并发编程能力 | worker 线程 pthread_cond_timedwait 阻塞等待，调度线程 mark_done 唤醒 |
| **并发限流** | 展示生产系统思维 | 信号量替代忙等轮询，batch 槽位上限控制 GPU 并发 |
| **超时控制** | 防止资源泄漏 | deadline 过滤 + cleanup_expired，毫秒级精度 |
| **优先级调度** | 展示灵活性设计 | 客户端 JSON 指定优先级，高→中→低队列出队 |
| **抽象接口解耦** | 展示设计模式认知 | `IInferenceEngine` 纯虚接口，MockEngine 和 ONNX Runtime 互换 |
| **无框架依赖** | 展示底层功底 | epoll + HTTP/1.1 手写解析 + pthread 线程池，没调任何框架 |
| **跨版本适配** | 展示工程能力 | ONNX Runtime 1.19 API 签名变更，读头文件源码适配 |

### 2.3 面试中可能被问到的问题

**Q: 为什么不用现成的推理服务框架（Triton、TorchServe）？**

A: 这个项目的目的是学习底层原理。用现有框架不需要理解调度、批处理、线程同步，而这个项目让你看到了"框架底层在做什么"。同时也说明你有能力在需要特殊优化场景下定制方案。

**Q: 为什么用 C++ 而不是 Python？**

A: 推理网关对延迟和并发要求高——C++ 无 GC 停顿、可直接管理内存、epoll 网络模型效率高。Python 写推理网关会遇到 GIL 瓶颈，且 http.server 性能远不如 epoll。

**Q: 动态批处理和静态批处理有什么区别？**

A: 静态批处理要求客户端显式打包好 batch 再提交——客户端自己等。动态批处理是服务端透明的——客户端各自独立发请求，服务端攒够一批再推理。服务端攒 batch 的延迟（batch_window_ms）远小于客户端各自等待的延迟。

**Q: 如果队列里不断有新任务进来，低优先级任务会不会永远得不到处理？**

A: 不会。collect_batch() 先从高优队列取，取完 max_batch_size 后再从低优队列取。高优任务只决定进入 batch 的顺序，不垄断资源。再加上并发 batch 槽位，低优任务最终都会被处理。

**Q: onnx 模型加载失败怎么办？**

A: 自动回退到 MockEngine，服务不会挂。stderr 和日志会打印错误信息。

**Q: 这个项目的实际性能怎么样？**

A: 在 MockEngine 模拟 100ms 延迟下，顺序 QPS 8.1 → 高并发 QPS 64.3，加速比 7.9x。延迟分位数稳定（P99 仅比均值高 ~3%）。

## 三、项目现状与限制

### 已完成功能

- [x] 完整链路（HTTP 解析 → 调度 → 推理 → 响应）
- [x] 动态批处理调度（三级优先级 + 积累窗口）
- [x] 超时控制（毫秒级）
- [x] 优先级调度（客户端指定）
- [x] 并发限流（batch 槽位 + 队列上限 + 503）
- [x] 压测对比（7.9x 加速比）
- [x] ONNX Runtime 真实推理
- [x] 10 类场景 28 个测试用例

### 已知限制（面试时诚实说明）

| 限制 | 影响 | 说明 |
|---|---|---|
| 测试模型是假的 | 没有真实模型推理效果 | 替换 .onnx 文件即可，架构完全支持 |
| JSON 解析是手写的 | 只支持当前字段格式 | 不支持嵌套、不报解析位置。后续可换 nlohmann/json |
| usleep 积累窗口 | 低负载下单请求延迟增加 batch_window_ms | 可改为基于时间的动态等待 |
| pthread 写同步原语 | 不如 std::mutex/std::condvar 安全 | 有析构 throw warning，不影响运行 |
| 继承了大量 web_server 代码 | 有约 60% 代码与推理网关无关 | 后续可精简 |

## 四、未来改进方向

### 短期（1-2 周，投入产出比最高）

#### 4.1 换真实 ONNX 模型

去 [ONNX Model Zoo](https://github.com/onnx/models) 下载一个视觉模型（ResNet-50、MobileNet-v2），放项目目录，启动时指定路径。这让"AI 推理网关"名副其实。

#### 4.2 性能对比图表

运行 `benchmark.py` 不同参数组合，收集数据画图表：

```
batch_size:  1, 2, 4, 8, 16, 32
concurrency: 1, 2, 4, 8
engine_latency: 10ms, 50ms, 200ms

输出: QPS 对比表 + 折线图
```

对面试极有用——能用图表展示"动态批处理到底提升多少吞吐"。

#### 4.3 加性能监控接口

```
GET /health    → {"status":"ok","pending":5,"active_batches":2,"uptime":3600}
GET /stats     → {"total_requests":12000,"avg_latency_ms":56,"qps":142}
```

面试时能说"暴露了监控端点，可以接 Prometheus"。

### 中期（1-3 个月）

#### 4.4 用 libevent 或 asio 替换手写 epoll

当前 epoll 代码是手动管理 fd + EPOLLONESHOT + 定时器链表的组合。换个成熟的网络库可以让代码量减半，同时学会工业级网络编程。

#### 4.5 JSON 解析从手写换成 nlohmann/json

`#include <nlohmann/json.hpp>` 一个头文件。支持嵌套、验证、清晰错误提示。顺便展示你会选择何时引入依赖、何时自己造轮子。

#### 4.6 支持流式响应（SSE）

大模型推理场景（LLM）是逐个 token 输出的。支持 `Transfer-Encoding: chunked` 或 SSE 推送，让客户端能实时看到推理输出。

#### 4.7 支持 GPU 推理

ONNX Runtime 支持 CUDA/CUDAExecutionProvider。加一个 `-g` 参数，调用方传入 `Ort::CUDAProviderOptions`。展示你会用 GPU 编程接口。

### 长期（架构层面）

#### 4.8 注册中心 + 多实例负载均衡

多个 ai_gateway 实例注册到 etcd/Consul，前置 nginx 做负载均衡。面试时能展开讲分布式系统。

#### 4.9 请求缓存

相同输入（hash 判重）直接返回缓存结果，跳过推理。对推荐系统、搜索等场景意义大。

#### 4.10 模型热加载

不重启服务就能切换模型。用 inotify 监听模型文件变化，或者 gRPC 管理接口触发。

## 五、面试注意事项

### 5.1 怎么介绍这个项目（3 分钟版本）

> 我做了一个 C++ AI 推理服务网关。核心问题是单实例模型无法服务多客户端并发请求，我设计了动态批处理调度器——把多个独立请求在服务端透明合并成一批推理，实测吞吐提升 8 倍。
>
> 整个项目从网络层到推理层都是自己写的，没调框架。网络层用 epoll + 手写 HTTP 解析，调度层有三层优先级队列、积累窗口、超时控制和并发限流，引擎层用抽象接口解耦 Mock 引擎和真实 ONNX Runtime。
>
> 约 3700 行 C++，28 个端到端测试，支持命令行参数调参，有压测脚本可以跑加速比数据。

### 5.2 不要主动说的

- **"我是跟着教程做的"** —— 这会给面试官一种"复制粘贴"的印象。你是自己理解需求、设计方案、实现的。
- **"这部分我不太懂"** —— 如果你对某个模块理解不深，先说自己确定的，不确定的放到后续讨论。不要说"不太懂"就结束。
- **"其实没什么难度"** —— 贬低自己的劳动。这个项目的技术含量是客观存在的。

### 5.3 如果面试官追问暴露短板

| 追问 | 怎么答 |
|---|---|
| "epoll 和 io_uring 的区别？" | 诚实说没用过 io_uring，epoll 是事件通知模型，io_uring 是异步 IO 提交/完成环形队列，适合超高吞吐场景。表达有兴趣学习。 |
| "为什么不用 gRPC？" | 当前是 RESTful JSON API，简单通用。gRPC + protobuf 序列化更快、支持 streaming，适合大规模部署时升级。坦诚这是后续优化方向。 |
| "你这个代码有内存泄漏吗？" | 主要资源是 C 风格 new/delete，已知析构链正确。可以用 Valgrind 跑一遍验证（其实你没跑过，坦诚说）。 |

### 5.4 简历怎么写

```
AI 推理服务网关 | C++ | 个人项目

• 设计并实现基于 epoll 的高并发推理网关，支持动态批处理、优先级调度、超时控制
• 核心调度器将分散的推理请求自动合并为批量推理，实测吞吐提升 7.9 倍

• 技术栈：C++、epoll、pthread、ONNX Runtime、HTTP/1.1、Python（测试/压测）
• 代码量 ~3700 行，10 类场景 28 个端到端测试
```
